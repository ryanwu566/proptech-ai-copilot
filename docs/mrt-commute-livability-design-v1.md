# MRT 通勤與生活機能分析產品設計規格 v1

## 1. 產品目的

本功能放在 PropTech AI Copilot 的「通勤與生活機能」區塊，目的是協助使用者理解物件周邊的捷運通勤特徵，補充生活機能與交通可近性的判讀。

本功能只能提供通勤與生活機能參考，不得直接判斷：

- 值不值得買
- 值不值得看房
- 房價高低或增值潛力
- 地勢災害風險
- 法律結論
- 貸款結論
- 稅費結論
- 文件真偽或合規結論

設計上應採保守敘述，避免把通勤觀察包裝成投資建議或安全判定。

## 2. 參考的分析邏輯

本產品保留以下捷運分析邏輯，作為未來資料層與介面設計依據：

- 站點活力：觀察站點是否具有穩定且明顯的人流存在感
- 平日／假日節律：比較工作日與週末的使用型態差異
- 尖峰壓力：觀察早晚尖峰是否集中、是否有明顯壅塞特徵
- 轉乘角色：辨識站點是否扮演轉乘、集散或中繼角色
- 路網可達性：理解站點在整體路網中的連通性與到達便利性
- 歷史 OD／旅次模式：觀察 origin-destination 旅次分布與方向性
- 關鍵站與區域連結：辨識對跨區通勤、核心商業區或生活圈連結較強的站點

上述邏輯僅用於通勤與生活機能描述，不應直接推導使用者的購屋決策。

## 3. 動態資料原則

本功能必須將「即時／近期資料」與「歷史分析資料」明確區分，並在每個結果上顯示資料日期、來源與資料限制。

### 3.1 資料呈現原則

- 每一項結果都要標示資料日期
- 每一項結果都要標示資料來源
- 每一項結果都要標示資料限制或可信度說明
- 若資料缺失，不可推測或補值為既定事實
- 若資料不足，必須顯示「資料不足，無法判定」
- 不可把缺資料誤讀為低人流、低壓力、低便利或低價值

### 3.2 歷史資料限制

- 2025 OD 資料只能作為歷史觀察
- 歷史資料不得被描述為目前即時狀態
- 歷史資料不得直接代表當前路網、班次或站點營運現況
- 若未確認可更新性，歷史資料只能作為趨勢參考

## 4. 建議未來資料層

下列資料層為未來建議架構。每層都應評估是否需要官方來源、更新頻率、資料限制，以及是否可能影響使用者結論。

### 4.1 靜態路網與站點資料

- 內容：站名、站序、線別、轉乘關係、地理位置、服務範圍
- 是否需官方來源：是，優先使用官方或可驗證來源
- 更新頻率：低頻，路網異動時更新
- 資料限制：需確認站名、轉乘與線別的版本一致性
- 是否可影響使用者結論：僅可影響通勤與生活機能描述，不可影響風險或財務結論

### 4.2 定期更新的站點運量資料

- 內容：站點進出站量、尖峰時段、平假日差異
- 是否需官方來源：是
- 更新頻率：建議月更或季更，依官方發布週期調整
- 資料限制：不同口徑可能無法直接比較
- 是否可影響使用者結論：僅可影響通勤與生活機能判讀

### 4.3 歷史 OD／旅次資料

- 內容：站點間旅次、流向、區域連結、旅次模式
- 是否需官方來源：是，並保留可驗證的資料說明
- 更新頻率：低頻或批次更新
- 資料限制：只能代表歷史區間
- 是否可影響使用者結論：僅可作為歷史觀察與通勤模式研究

### 4.4 地址定位後的最近站／步行接近性

- 內容：物件地址對應最近站、步行距離、步行時間等
- 是否需官方來源：建議採可驗證地圖與路網來源
- 更新頻率：依地址解析與地圖資料更新
- 資料限制：路徑、步行時間與入口位置可能有差異
- 是否可影響使用者結論：可影響通勤與生活機能摘要，不可影響風險或法律結論

### 4.5 未來可選的目的地通勤比較

- 內容：指定目的地如公司、學校、醫院、商圈的通勤比較
- 是否需官方來源：建議是，或至少可驗證來源
- 更新頻率：依目的地與路網資料更新
- 資料限制：不同交通模式與時段會改變結果
- 是否可影響使用者結論：僅可作為通勤參考，不可作為購屋決策結論

## 5. 首版 MVP

首版只做最小且保守的功能：

- 最近站
- 站點角色
- 資料日期
- 保守提醒

### 5.1 MVP 輸出內容

- 物件地址對應的最近捷運站
- 站點在通勤與生活機能上的簡短角色描述
- 資料日期與來源註記
- 保守提醒，例如「僅供通勤與生活機能參考」

### 5.2 MVP 不做的事

- 不做即時 ETA
- 不做票價計算
- 不做導航
- 不做地圖嵌入
- 不做外部網站嵌入
- 不做綜合分數
- 不做買房建議
- 不影響 Viewing Decision、Terrain Risk、稅費、貸款或文件結論

## 6. 未來分階段路線

### Phase 1：通勤資訊卡

目標：先提供最近站、站點角色與資料日期。

風險：

- 若站點資料不完整，容易產生誤導
- 若地址定位不準，最近站可能錯誤

驗證條件：

- 可穩定對應地址到最近站
- 可顯示明確的來源與日期
- 文案不會越界到投資判斷

### Phase 2：站點節律與尖峰觀察

目標：補充平日／假日節律、尖峰壓力與站點活力。

風險：

- 容易把歷史運量誤當即時狀態
- 容易把熱鬧程度誤讀為房產價值

驗證條件：

- 可清楚區分歷史與近期
- 可在資料不足時明示無法判定
- 不會輸出買賣結論

### Phase 3：歷史 OD／區域連結

目標：呈現歷史 OD、旅次模式、關鍵站與區域連結。

風險：

- 資料量與口徑差異可能造成誤讀
- 可能被使用者誤當作即時通勤現況

驗證條件：

- 有資料區間與來源標示
- 只作為歷史觀察
- 不影響其他模組結論

### Phase 4：指定目的地通勤比較

目標：讓使用者比較住家到公司、學校、醫院或商圈的通勤便利性。

風險：

- 需要更嚴謹的路徑與時間來源
- 容易擴張成完整導航功能

驗證條件：

- 可控的來源與更新週期
- 明確區分建議、參考與限制
- 不變成即時導航或交通監控工具

## 7. 使用者文案範例

### 7.1 通勤資訊卡

- 「此資訊僅供通勤與生活機能參考，資料來源為官方或可驗證來源，更新日期為 2026-xx-xx。」
- 「最近捷運站為 OOO，屬於通勤轉乘較方便的站點之一；實際步行時間仍會受入口位置與路線影響。」
- 「目前資料不足，無法判定此站的近期尖峰壓力，請以後續更新資料為準。」

### 7.2 站點節律

- 「此站平日使用明顯高於假日，顯示其通勤屬性較強，僅供通勤與生活機能參考。」
- 「本結果為歷史觀察資料，不能直接代表目前即時狀態。」

### 7.3 歷史 OD

- 「此區域在歷史 OD 資料中與核心站有明顯連結，僅供通勤與生活機能參考。」
- 「本結果來自歷史區間資料，不代表當前班次、車流或站點營運狀態。」

### 7.4 保守提醒

- 「僅供通勤與生活機能參考，不能作為購屋決策、風險判定或財務結論依據。」
- 「若資料不足，系統將顯示『資料不足，無法判定』。」

## 8. 產品邊界與明確排除功能

本規格明確排除以下能力：

- 不直接判斷值得買或值得看房
- 不直接判斷房價高低、增值潛力或投資報酬
- 不直接判斷災害風險、地勢風險或法律風險
- 不作為貸款、稅費、法務或文件結論來源
- 不做即時 ETA
- 不做票價
- 不做導航
- 不做地圖或第三方網站嵌入
- 不依賴無法驗證的第三方資料集
- 不掃描或猜測未公開 API
- 不把歷史資料包裝成即時資料

## 9. 建議的資料治理原則

- 優先採用官方或可驗證來源
- 來源、日期、區間、限制須可追溯
- 資料不足時寧可不判定，也不要推測
- 歷史資料與即時資料分層管理
- 每個輸出都要能說明「這只是通勤與生活機能參考」

## 10. 結語

本產品的核心不是判斷房產好壞，而是讓使用者更清楚理解物件周邊的捷運通勤特徵、生活機能連結與歷史旅次脈絡。所有輸出都應保守、可追溯、可驗證，並維持與 Viewing Decision、Terrain Risk、稅費、貸款與法律結論的明確隔離。
