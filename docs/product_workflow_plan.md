# PropTech AI Copilot 買房決策流程與後續五關計畫

更新日期：2026-06-08  
原則：每一關可獨立測試、可獨立部署，不新增登入、PDF、RAG 或房屋網爬蟲。

## 1. 目標產品流程

```text
找房雷達
  → 帶入估價
  → 查看可比成交與趨勢
  → 查看地圖區位與生活機能
  → 試算頭期款、月付與利率敏感度
  → 試算稅費與持有成本
  → 彙整資料信心、價格、流動性、貸款與稅務風險
  → 分享連結 / HTML 看屋決策摘要
```

### 跨模組連貫規則

1. 使用者選定的 city、district、road、building_type、area_ping、building_age_years、floor 與估價總價，應能在模組間沿用。
2. 任何帶入動作只填入條件，不自動送出付費或外部 API 查詢。
3. 每個結果都顯示資料來源、期間、信心與免責。
4. 官方資料、展示樣本、mock fallback 必須清楚區分。
5. 不宣稱找到即時待售物件，也不宣稱精準預測未來房價。

## 2. 五關 Roadmap

## 關卡 1：Data-status 效能驗證與 UI 收斂

### 目標

確認 45 萬筆以上官方 PLVR 下，`/valuation/data-status` 快速、穩定，前端只呈現簡短覆蓋摘要。

### 使用者價值

使用者能快速理解資料可信度與覆蓋，不會被行政區長文字淹沒，也不會因資料狀態卡拖慢估價頁。

### 後端

- 保留 `coverage_city_count`、`coverage_district_count`、`coverage_road_count`、`coverage_summary`、`coverage_note_short`。
- 保留 `official_coverage_note` 相容性，但限制長度。
- 量測 Postgres 聚合查詢時間；必要時加入 process cache 或輕量 summary cache。
- 確保 `/health` 不依賴 data-status 聚合。

### 前端

- 只顯示官方/樣本筆數、raw/effective period、rolling 3 年、覆蓋數與最近匯入。
- 詳細覆蓋改為折疊或文件連結，不渲染完整行政區清單。
- 確認手機版不爆版。

### 測試

- data-status 結構化欄位存在。
- `official_coverage_note` 有長度上限。
- 前端不直接渲染長 note。
- Postgres exception/fallback 仍回安全結構。
- 量測正式環境 response time。

### 部署

- Render：需要。
- Vercel：需要。

### 風險

- 即時計數在資料量持續增加時仍可能昂貴。
- cache 需要避免長時間顯示舊匯入狀態。

## 關卡 2：找房雷達 Property Finder v1 穩定化

### 目標

把目前已存在的找房雷達正式收斂成「依預算與條件找歷史成交方向」，並順暢帶入估價與地圖。

### 使用者價值

回答「我的預算大概可以看哪些區域與路段」，再進入合理價格與生活圈判讀。

### 後端

- 穩定 `POST /valuation/property-search`。
- 只使用官方 PLVR rolling 3 年。
- 保留可解釋分數：預算接近度、成交量、近期資料。
- 增加安全 query limit 與空結果說明。
- 可選新增「候選路段地圖查詢字串」，不改核心估價 schema。

### 前端

- 查詢條件與結果摘要。
- 推薦行政區、路段與歷史成交。
- 「帶入估價」與「查看區域地圖」CTA。
- 明確標示不是即時待售物件。
- 手機表格區塊左右滑，不造成整頁 overflow。

### 測試

- 預算、city、district、type、period 篩選。
- 不使用 sample/mock，排除未來期間。
- limit 最大 100。
- 帶入估價不自動送出。
- 空結果與手機版。

### 部署

- Render：需要。
- Vercel：需要。

### 風險

- 使用者可能誤解為待售物件推薦。
- 全資料範圍搜尋可能增加 Postgres 負載，需限制候選池。

## 關卡 3：貸款月付試算 v1

### 目標

用透明公式回答頭期款、貸款金額、每月本息、收入負擔率與利率變動影響。

### 使用者價值

將「這間房合理價多少」轉成「我每月付不付得起」。

### 後端

- 新增純計算 service/API，不依賴銀行核貸模型。
- 輸入：總價、頭期款或貸款成數、年利率、年限、寬限期、月收入、既有月負債。
- 輸出：貸款金額、月付、總利息、負擔率、利率 ±0.5% / ±1% 敏感度。
- 可使用央行/銀行牌告利率作預填參考，但不代表核貸利率。

### 前端

- 從估價總價一鍵預填。
- 顯示頭期款、一般月付、寬限期後月付、負擔率與敏感度。
- 風險提示使用自然文案，不只顯示 heuristic 分數。
- 保留 Aegis-Credit 展示風險，改為次要區塊。

### 測試

- 本息平均攤還公式。
- 零利率、不同年限、不同頭期款。
- 利率敏感度。
- 非法輸入與極端值。
- 不宣稱核貸保證。

### 部署

- Render：需要。
- Vercel：需要。

### 風險

- 寬限期與不同還款方式容易造成誤解。
- 必須明確標示實際條件依銀行審核。

## 關卡 4：地圖區位實用化 v1

### 目標

讓 Map Insight 不只是地圖上的點，而是可比較、可帶回案件的區位優缺點摘要。

### 使用者價值

回答「這個路段住起來方便嗎、有哪些優勢與注意事項」，補充價格以外的決策。

### 後端

- 延用 Google Places / mock fallback 與 800m 評分。
- 輸出明確的 `strengths`、`watchouts`、最近重要設施與資料來源。
- 可支援兩至三個候選路段的簡單比較。
- 不引入黑箱估價加分；地圖分數不直接改變房價估值。

### 前端

- 找房雷達與估價結果增加「查看此路段地圖」。
- 顯示區位優點、注意事項、最近設施與六類指標。
- 候選路段比較使用 compact cards。
- 保留地圖主視覺、圖層與資料來源 badge。

### 測試

- Google / mock fallback。
- 同一案件路段可正確帶入。
- strengths/watchouts 有可解釋依據。
- API 錯誤不破壞頁面。
- 手機版地圖在上、摘要在下。

### 部署

- Render：需要。
- Vercel：需要。

### 風險

- Google quota、資料更新頻率與 geocoding 偏差。
- POI 數量不等於實際生活品質，文案需保守。

## 關卡 5：看屋決策摘要 v2

### 目標

把找房、估價、趨勢、地圖與貸款結果整合成一份可分享的 HTML 看屋決策摘要。

### 使用者價值

家人、客戶或房仲可在同一份摘要中理解價格、區位、負擔與主要風險。

### 後端

- 初期可維持前端 Blob HTML，不新增 PDF。
- 若需要一致格式，再新增 report DTO / service，但不保存敏感個資。
- TaxOracle 報告保持獨立章節，可選擇納入。

### 前端

- 報告章節：條件、找房方向、估價、趨勢、地圖、貸款、主要風險、資料來源與免責。
- 分享連結仍只帶條件，明確提示結果需重新查詢。
- 增加報告預覽與資料更新時間。

### 測試

- 有/無各模組結果時皆能匯出。
- HTML escape 與長文字。
- 手機瀏覽與列印樣式。
- 免責、來源與資料期間完整。
- TaxOracle 原有 HTML report 不受影響。

### 部署

- Render：視 report DTO 是否新增；若只改前端則不需要。
- Vercel：需要。

### 風險

- 報告過長、不同結果時間不一致。
- 分享條件與結果快照的語意需清楚。

## 3. 後續候選關卡

上述五關完成後，再評估：

1. 買房稅費與持有成本 v1：契稅、登記、仲介、房屋稅/地價稅情境與年度持有成本。
2. 行政區比較 v1：以官方 PLVR 取代 Market Insight mock 趨勢。
3. 統一案件狀態：讓 History 從 TaxOracle 歷史擴展為跨模組案件進度。
4. 風險摘要 v2：整合資料信心、開價偏離、流動性、波動、貸款與稅務。
5. 中文編碼與前端元件拆分：獨立品質關卡，不與產品功能混做。

## 4. 建議下一關

若以使用者價值與目前成熟度排序，建議下一關做：

> **貸款月付試算 v1**

理由：

- Data-status 與 Property Finder 的核心實作已存在，適合先做正式環境穩定化，不必再擴大功能。
- 估價已能提供合理總價，央行與銀行牌告利率也已存在；月付試算能直接把兩者連起來。
- 這是目前主流程中最明顯、對一般買方最有感的缺口。
- 可用透明公式實作，風險低、可測試、無需新增外部 API 或資料庫。
